# 模型服务优化对比分析

## 1. 方案对比总览

| 对比项 | vLLM 加速方案 | 原方案（双模型加载） | 优化方案（统一VL服务） |
|--------|--------------|-------------------|---------------------|
| **状态** | ❌ 无法运行 | ⚠️ 可运行但效率低 | ✅ 运行正常 |
| **GPU 显存需求** | 32GB+ | ~30GB（峰值） | ~23GB |
| **可用性** | RTX 3090 24GB 显存不足 | 可运行但需交替加载 | 完全可用 |
| **模型加载次数** | 1次 | 2次 | 1次 |

## 3. 详细性能对比

### 3.1 vLLM 加速方案（失败）

| 指标 | 数值 | 说明 |
|------|------|------|
| **状态** | ❌ 失败 | 无法加载模型 |
| **错误类型** | OOM | GPU 显存不足 |
| **尝试配置** | max_model_len=4096 | 最小上下文长度 |
| **gpu_memory_utilization** | 0.70-0.85 | 多次尝试均失败 |
| **主要错误** | `KV cache memory needed larger than available` | vLLM V1 引擎需要更多显存 |
| **结论** | 需要 32GB+ 显存 | RTX 3090 无法满足 |

### 3.2 原方案（双模型加载）

| 指标 | 数值 | 说明 |
|------|------|------|
| **模型加载** | 2次独立加载 | OCR服务 + LLM服务各加载一次 |
| **服务架构** | 分离式 | `qwen_vl_service.py` + `llm_service.py` |
| **GPU 显存使用** | ~15GB × 2（交替） | 无法同时加载两个模型实例 |
| **内存 RSS（峰值）** | ~4600MB | 根据对话摘要数据 |
| **OCR 耗时** | ~25-30秒 | 包含模型加载时间 |
| **LLM 分析耗时** | ~15-20秒 | 可能需要重新加载模型 |
| **总处理时间** | ~60-80秒 | 含模型切换开销 |
| **模型切换** | 需要 | OCR 和 LLM 交替使用 GPU |
| **并发能力** | 低 | 模型切换导致延迟 |

### 3.3 优化方案（统一VL服务）

| 指标 | 数值 | 说明 |
|------|------|------|
| **模型加载** | 1次 | 单例模式共享模型实例 |
| **服务架构** | 统一式 | `unified_vl_service.py` |
| **GPU 显存使用** | ~23GB | 23299MB / 24576MB |
| **内存 RSS（稳定）** | ~2058MB | 减少 55% |
| **OCR 耗时** | 13-27秒 | 无模型加载开销 |
| **LLM 分析耗时** | 4-15秒 | 复用已加载模型 |
| **总处理时间** | ~18-43秒 | 减少约 50% |
| **模型切换** | 不需要 | OCR 和 LLM 共享模型 |
| **并发能力** | 高 | 无模型切换延迟 |

## 7. 关键优化点

1. **单例模式**：使用 `__new__` 方法确保模型只加载一次
2. **延迟加载**：首次使用时才加载模型
3. **共享实例**：OCR 和 LLM 共享同一个模型和处理器
4. **配置开关**：保留 `USE_VLLM` 配置，便于未来升级到更大显存的 GPU

## 9. 总结

| 优化维度 | 提升效果 |
|---------|---------|
| 内存使用 | 减少 55% |
| 处理时间 | 减少 50%（同尺寸图片） |
| GPU 利用率 | 提高（无切换开销） |
| 并发能力 | 显著提升 |
| 稳定性 | 支持高分辨率图片，无 OOM |
| 代码复杂度 | 降低 |

**结论**：
1. 统一 VL 服务方案在 RTX 3090 24GB 环境下是最优选择
2. Flash Attention 2 需要较长编译时间，建议后续安装
3. 图像预处理有效防止高分辨率图片 OOM
4. 未来升级到更大显存 GPU 后可启用 vLLM 加速
